# V2 Per-result Rule Version Propagation Evidence

## 1. Purpose

V2 Phase 12 extends Phase 11 ruleVersion alignment from contract-level evidence to detection-result-level traceability.

The goal is not fraud detection performance improvement, threshold tuning, or a new rule baseline. The goal is to prove that newly generated detection results can carry the active app-consumer `ruleVersion`, and that PaySim evaluation can distinguish legacy exports from per-result verified exports.

README remains an entry point only. Detailed commands, troubleshooting, and verification results live in docs, blog drafts, and `scripts/data/README.md`.

## 2. Problem

Report-level `ruleVersion` proves which evaluation contract was selected, but it does not prove that every detection result row was generated by the same rule version.

That gap matters when:

- rule deployment changes during replay/export windows
- a detection result export mixes old and new rows
- `ruleVersion` and `thresholdVersion` are interpreted as one version dimension
- legacy exports omit per-result `ruleVersion`
- fixture checks accidentally treat missing values as a distribution key

Phase 12 keeps backward compatibility for legacy rows while adding a strict mode for stronger evidence.

## 3. Completed Scope

Implemented scope:

- `FraudRuleEngineResult` now includes `ruleVersion`.
- `FraudRuleEngine` sets `FraudRuleVersions.ACTIVE_RULE_VERSION` on each result.
- `fraud_detection_results.rule_version` is added with a nullable migration for compatibility.
- app-consumer saves `ruleVersion` for newly generated fraud detection results.
- app-api admin fraud result response exposes nullable `ruleVersion`.
- The API module remains the Flyway migration owner. app-consumer runs with JPA validate and expects app-api migrations, including `rule_version`, to be applied before consumer startup.
- Python evaluator report schema is bumped to `2026-06-v2-phase12`.
- Python evaluator adds `--require-per-result-rule-version`.
- Default evaluation mode keeps legacy compatibility with warning and coverage fields.
- Strict per-result mode fails when evaluated result rows omit `ruleVersion`.
- `verify-paysim-result-rule-version-contract` validates present, missing, mixed, mismatch, and strict-mode fixtures.
- `verify-v2-phase12` and `final-check` include the new CI-safe verifier.

Not implemented in this phase:

- historical backfill for old `fraud_detection_results` rows
- automated DB detection result export generation
- new fraud rule logic or threshold tuning
- full PaySim replay/evaluation output commit

## 4. Version Field Semantics

| Field | Scope | Owner | Required For New Results | Missing Handling | Verification |
|---|---|---|---|---|---|
| `ruleVersion` | rule logic baseline | app-consumer | yes for newly generated detection results | legacy warning by default, strict fail when required | `verify-paysim-result-rule-version-contract` |
| `thresholdVersion` | evaluation threshold boundary | scripts/data | report-level | fail if unknown | `verify-paysim-rule-threshold-regression` |
| `mappingPolicyVersion` | PaySim native mapping | scripts/data | report-level | fail if mismatch | `verify-paysim-native-replay-contract` |
| `evaluationContractVersion` | denominator/missing/excluded policy | scripts/data | report-level | fail if unknown | `verify-paysim-evaluation-report-contract` |

Phase 12 updates `reportSchemaVersion` for per-result ruleVersion readiness and strict-mode reporting fields. It keeps `evaluationContractVersion` unchanged because denominator and missing-result evaluation semantics did not change.

## 5. Backward Compatibility Policy

Legacy detection result exports may not include `ruleVersion`.

Default evaluator behavior:

- missing per-result `ruleVersion` is allowed
- missing rows are counted in `ruleVersionCoverage`
- a warning explains that evaluation falls back to contract-level `ruleVersion`
- `ruleVersionDistribution` includes only present per-result values

Strict evaluator behavior:

- `--require-per-result-rule-version` requires every evaluated result row to include expected `ruleVersion`
- missing per-result `ruleVersion` fails before report generation
- mismatched per-result `ruleVersion` always fails

Readiness values:

| Value | Meaning |
|---|---|
| `contract_level_only` | evaluated rows do not provide per-result `ruleVersion` |
| `contract_level_with_partial_per_result_coverage` | some evaluated rows provide `ruleVersion`, some rely on legacy fallback |
| `per_result_verified` | every evaluated result row has the expected `ruleVersion` |

## 7. Failure Modes

- report-level `ruleVersion` is overclaimed as row-level consistency
- legacy export fails strict mode because per-result `ruleVersion` is absent
- mixed present/missing rows create a `null` or `None` distribution key
- mismatched per-result `ruleVersion` is treated as a warning instead of fail-fast
- Java/Python active ruleVersion drift is missed
- `reportSchemaVersion` is not bumped after adding report fields
- persistence schema changes but API/export DTOs do not expose the field
- `ruleVersion` and `thresholdVersion` are reviewed as one version dimension

## 8. Limitations

- PaySim is a synthetic dataset.
- Per-result `ruleVersion` is traceability evidence, not a fraud performance guarantee.
- A stored `ruleVersion` does not prove the rule logic is correct.
- Historical rows may have null `rule_version`.
- Full PaySim replay and DB export evidence remain local/manual.
- Automated detection result export generation remains future work.
